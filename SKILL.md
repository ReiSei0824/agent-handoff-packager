# 代理交接包设计师（Agent Handoff Packager）

## 核心定位

这个 skill 解决的不是“怎么再写一个提示词”，而是“怎样把一项要交给 AI 代理的任务交接清楚”。当用户准备把分析、写作、代码、研究或运营工作交给 ChatGPT Work、Codex、Claude Code、Gemini 或类似代理时，它会先把任务边界、材料、权限、风险和验收标准整理成一份可直接发送的 handoff pack。

它负责交接设计、交付标准和复核节点；不负责替用户跳过判断，也不替用户把本来不适合委派的任务强行外包给代理。

---

## 触发场景

- 用户想把一个多步骤任务交给 AI 代理，但担心返工、跑偏、漏文件或误操作。
- 用户说“帮我整理一份交给 ChatGPT Work / Codex / Claude Code 的任务包”“先别执行，先写交接说明”。
- 用户已有任务目标和材料，但缺少可复用的 brief、验收标准、停机线和复核节奏。
- 用户在复盘“为什么 agent 总做偏”时，希望先把交接问题拆开，而不是继续堆提示词。

---

## 工作流

### Step 1：判断这件事值不值得委派

先判断任务是否适合交给代理。优先识别三类不适合直接外包的情况：目标本身含糊、关键判断必须实时拍板、或者执行动作存在高风险且缺少回滚手段。若不适合，明确写出“建议人工先定目标 / 先缩 scope / 先补材料”，不要硬拼一个漂亮但不可执行的交接包。

### Step 2：盘点目标、材料和执行边界

把用户输入拆成五类信息：目标结果、现有材料、可访问文件或链接、不可触碰边界、时间与成本约束。若材料缺失但还能继续，就把缺口标成“待补输入”；若缺口会直接影响执行正确性，就在交接包里把对应步骤改成“先确认再继续”。

### Step 3：定义完成标准和停机线

为代理写清楚 done definition，而不是只写“做完就行”。至少覆盖：要交付什么、什么算通过、哪些情况必须停下汇报、哪些内容只能建议不能直接改动。涉及代码、数据、合规或对外发布时，要额外补充验证动作、回滚办法和人工复核点。

### Step 4：设计执行节奏与检查点

把任务拆成 3-6 个阶段，每个阶段写明输入、动作、输出和检查条件。若任务跨度大于一次性执行范围，要求代理先交中间稿或计划，再进入下一阶段。若用户说“直接干”，也要在交接包里保留最小检查点，避免代理长时间无反馈地越权扩张。

### Step 5：组装可直接发送的 handoff pack

输出一份结构化 Markdown 交接包，默认包含：任务摘要、目标成果、背景上下文、必读材料、可用文件、禁止事项、验收标准、汇报节点、失败兜底和一段可直接复制给代理的发送版提示。若用户指定目标平台，还要把语气和格式调成该平台最自然的版本。

### Step 6：自检可执行性

在交付前检查四个问题：代理能否仅凭这份包启动、是否有缺少的文件或链接、是否存在模糊动词如“优化一下”、是否把用户本应保留的判断错误外包给模型。发现问题就直接改，不把这些坑留给用户自己补。

示例触发结果应当是“拿来就能发给代理”的交接包，而不是只给一段抽象建议。

## Gotchas

1. **交接包不是长提示词堆砌**
   如果一份 handoff pack 只是把用户原话扩写三倍，却没有文件、边界、验收和停机线，它仍然不是可执行交接。长度不是质量，结构才是。

2. **模糊目标会放大代理返工**
   “帮我整理一下”“优化一下”“做成更专业的版本”这类说法必须落成可判断结果，否则代理只会在错误方向上更努力。

3. **不要把关键判断默认为模型来拍板**
   涉及取舍、品牌口径、对外承诺、法律财务医疗风险时，交接包必须写清哪些点需要人工确认，不能因为代理“看起来能做”就默认交出去。

4. **文件可达性比背景解释更重要**
   很多失败不是因为代理不聪明，而是因为读不到正确文件、版本或链接。文件路径、权限范围和可用材料必须先于“补充说明”写清楚。

5. **检查点过少会让错误积累到最后**
   用户常说“做完再给我看”，但多步骤任务里这往往意味着一次性返工。交接包至少要有一个中间检查点，尤其是任务跨度超过半小时人工工作量时。

6. **不要把一次性任务伪装成通用流程**
   若材料、目标和判断标准都高度依赖单次上下文，就应该写成一次性交接包，而不是硬包装成长期 SOP。这个 skill 生成的是 handoff pack，不是假装万能模板。
